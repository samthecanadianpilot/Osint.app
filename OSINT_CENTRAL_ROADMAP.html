03 — Reference

Globe configuration.

The heart of the product. Copy-paste starting point for the react-globe.gl layers.

GlobeView.jsx
<Globe
  globeImageUrl="//unpkg.com/three-globe/example/img/earth-night.jpg"
  backgroundColor="#000000"
  atmosphereColor="#0071e3"   // soft blue atmosphere
  atmosphereAltitude={0.18}

  // ── animated trails ──
  arcsData={arcs}
  arcColor={d => d.color}
  arcDashLength={0.4} arcDashGap={0.2}
  arcDashAnimateTime={2000}

  // ── surface blips ──
  pointsData={visiblePoints}
  pointColor={d => d.color}
  pointAltitude={d => d.alt}
  onPointClick={p => select(p.id)}

  // ── ripple on selection ──
  ringsData={selected ? [selected] : []}
  ringColor={() => "#0071e3"}
  ringMaxRadius={4} ringPropagationSpeed={2}
/>

// auto-rotate after mount
globeRef.current.controls().autoRotate = true;
globeRef.current.controls().autoRotateSpeed = 0.4;

05 — Deployment

GitHub → Vercel.

One repo, pushed to GitHub, auto-deployed by Vercel. The catch: Vercel runs serverless functions, so the live WebSocket server can't live there. The plan adapts in two stages.

Now — Path B · Client-side sim RECOMMENDED

The seed data + geo.js motion run in the browser. Vercel hosts a pure static React app — one push, zero servers, $0. Looks fully live for a single viewer.

vercel · static · free

Later — Path A · Split hosting

When real feeds land (Phase 7), the Express + WebSocket server moves to Railway / Render / Fly.io (free tier, true sockets). Frontend on Vercel connects to its wss:// URL.

vercel + railway · live push

The Node backend is never thrown away — under Path B it's the local dev server and the home for real-data integration; under Path A it becomes production. Same code, two destinations.
